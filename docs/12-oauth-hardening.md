# 12 - OAuth/OIDC hardening model

This package now separates two security surfaces:

1. Provider token validation.
2. Redirect-flow transaction validation.

A provider token can prove that a provider knows a user. It does not automatically prove that the callback belongs to the login transaction initiated by the same browser, device, tenant or channel.

## Hardened redirect flow

```text
GET /api/auth/federated/{provider}/redirect
        |
        v
Create one-time OAuthAuthorizationState
        |
        +--> state
        +--> nonce for OIDC-capable providers
        +--> PKCE code_verifier / code_challenge for OIDC providers
        +--> tenant, user_type, channel, guard
        +--> optional user-agent/IP fingerprint
        |
        v
Redirect to provider
        |
        v
Provider callback
        |
        v
Consume and validate state once
        |
        v
Restore AuthContext from consumed state
        |
        +--> tenant_id
        +--> user_type
        +--> channel
        +--> guard
        +--> redirect_uri
        |
        v
Exchange authorization code
        |
        v
Validate id_token issuer, audience, authorized party and nonce
        |
        v
Create ExternalIdentity
        |
        v
Local user resolution / provisioning / role mapping / token issuing
```

## Native token flow

For `POST /api/auth/federated/{provider}/token`, the package preserves which provider token field the client submitted:

```json
{ "id_token": "..." }
```

or:

```json
{ "access_token": "..." }
```

This matters for OIDC providers such as Keycloak, Auth0, Okta or Azure AD:

| Submitted field | Behavior |
|---|---|
| `id_token` | Validate and decode as an OIDC ID token through JWKS. |
| `access_token` | Use `userinfo_endpoint` as a bearer access token when configured. |
| raw/unknown token type | JWT-looking tokens fall back to ID-token handling; otherwise userinfo is used when available. |

This prevents a native app's `id_token` from being sent to the `userinfo_endpoint` as if it were an access token.

## Configuration

```php
'security' => [
    'oauth_state' => [
        'enabled' => true,
        'ttl_seconds' => 300,
        'cache_prefix' => 'federated-auth:oauth-state:',
        'bind_user_agent' => true,
        'bind_ip' => false,
    ],

    'pkce' => [
        'enabled' => true,
        'method' => 'S256',
    ],

    'oidc' => [
        'nonce_enabled' => true,
        'require_azp_when_multiple_audiences' => true,
    ],

    'redirects' => [
        'allowed_hosts' => 'api.example.com,example.com',
        'allow_http_localhost' => false,
    ],
]
```

## Why state is one-time

The `state` value is stored in the configured cache and consumed with `Cache::pull()`.

That means:

- a callback without state is rejected;
- an expired state is rejected;
- a reused state is rejected;
- a state generated for another provider is rejected;
- a state generated by another browser/user-agent is rejected when fingerprint binding is enabled.

## Why callback context is restored from state

Provider callbacks normally return only:

```text
code
state
```

They do not usually return the original application context:

```text
tenant_id
user_type
channel
guard
redirect_uri
```

For multi-tenant applications, those values must be restored from the consumed `OAuthAuthorizationState` before the broker resolves, creates or touches an identity link. Otherwise a callback login could run without tenant scope.

The broker therefore consumes the state before calling the provider callback adapter and passes an enriched `AuthContext` through the rest of the login pipeline. OIDC adapters reuse that restored state for redirect URI, PKCE `code_verifier` and nonce validation.

## PKCE

For generic OIDC, Keycloak and Apple, the package generates:

```text
code_verifier
code_challenge = BASE64URL(SHA256(code_verifier))
code_challenge_method = S256
```

The authorization request includes the challenge and the token request includes the verifier.

Socialite-based providers such as Google/Facebook keep their standard Socialite token exchange. The package hardens those redirect flows with state validation. For provider-specific PKCE with Socialite, use a dedicated adapter that controls the code exchange.

## Nonce

OIDC-capable providers receive a transaction-specific nonce in the authorization request. When an `id_token` is returned, the adapter validates that the token nonce matches the stored transaction nonce.

## Redirect URI guard

If `FEDERATED_AUTH_ALLOWED_REDIRECT_HOSTS` is configured, dynamic redirect URIs are only accepted when the host is explicitly listed.

Recommended production settings:

```env
FEDERATED_AUTH_ALLOWED_REDIRECT_HOSTS=api.example.com,app.example.com
FEDERATED_AUTH_ALLOW_HTTP_LOCALHOST_REDIRECTS=false
```

For local development:

```env
FEDERATED_AUTH_ALLOWED_REDIRECT_HOSTS=localhost,127.0.0.1
FEDERATED_AUTH_ALLOW_HTTP_LOCALHOST_REDIRECTS=true
```

## Token storage

Provider tokens are still not stored by default:

```php
'identity_store' => [
    'store_provider_tokens' => false,
    'encrypt_provider_tokens' => true,
]
```

Only enable provider token storage when the application must call provider APIs after login. For pure login, store only the provider identity link.

## Safe response payload

The response no longer needs to expose the complete Eloquent model. Configure the allowed user fields:

```php
'response' => [
    'include_user' => true,
    'user_fields' => ['id', 'uuid', 'name', 'email', 'user_type', 'status_id', 'avatar'],
    'include_external_identity' => false,
]
```

This avoids accidentally leaking internal columns such as password hashes, remember tokens or operational flags.
